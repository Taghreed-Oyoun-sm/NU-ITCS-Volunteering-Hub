# NU ITCS Volunteering Hub – Use Case Document

## 1. Introduction
This document describes all system use cases for the NU ITCS Volunteering Hub.  
It specifies the system actors, use case descriptions, and interaction flows.

---

## 2. System Actors
- **Student** – can view all posts, request help, volunteer, receive endorsements.
- **TA/Dr** – can create academic opportunities and recommend students. Cannot view student posts.
- **Club Representative** – can create event volunteering posts and recommend students. Cannot view student posts.
- **Admin (Moderator)** – full access, manages posts, reports, and system content.

## 4. Detailed Use Case Descriptions

---

### **UC3 – Create Help Request**
**Actor:** Student  
**Preconditions:**  
- User is logged in  
- Student profile is completed  

**Main Flow:**  
1. Student selects “Create Help Request”.  
2. Student fills in title, description, tags (#DSA, #Math).  
3. System validates the information.  
4. Request is submitted for admin approval.  
5. Admin approves → post becomes public.  

**Alternative Flow:**  
- Admin denies request → student receives notification.

**Postconditions:**  
- Help request is visible to all users.

---

### **UC10 – Endorse Student**
**Actor:** TA/Dr  
**Preconditions:**  
- TA is logged in  
- Student exists in system  

**Main Flow:**  
1. TA searches for a student.  
2. Selects “Add Endorsement”.  
3. Chooses a skill area (DSA, OOP, etc.).  
4. System saves the endorsement.  
5. Student receives notification.  

**Postconditions:**  
- Endorsement is linked to student profile.

---

### **UC11 – Approve/Deny Posts**
**Actor:** Admin  
**Main Flow:**  
1. Admin opens “Pending Posts”.  
2. Chooses Approve or Deny.  
3. If approved → post becomes visible to allowed users.  
4. If denied → user receives feedback.
